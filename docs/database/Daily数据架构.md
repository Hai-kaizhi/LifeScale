# Daily 数据架构

> 本文说明 LifeScale 的 Daily 结构化数据模型与沉淀分层机制。本应用为**纯本地单机**：无云端、无同步，所有数据在本机 SQLite 与本地 Markdown 文件之间流转。

## 一句话核心

**当天结构化数据存本地 SQLite（真相源），沉淀后转写为纯净 Markdown 文件（沉淀产物）**。SQL 是热的、可变的；`.md` 是冷的、不可变的归档。

## 数据温度分层

| 温度 | 载体 | 特征 |
|---|---|---|
| 热（当天） | 本地 SQLite | 可变、结构化、CRUD 频繁 |
| 温（近期回看） | 本地 SQLite + `.md` | 双轨：实体可改，`.md` 快照 |
| 冷（历史归档） | 纯 `.md`（本地 Vault） | 不可变，本地 Markdown 文件 |

**关键约束**：当天数据未沉淀时，Vault 侧**没有当天的 `.md` 文件**。只有沉淀后才生成。这样当天数据始终留在 SQLite 自由编辑，沉淀瞬间冻结为归档。

## 数据模型

### 本地 SQLite（唯一真相源）

当天结构化实体，详见 [`schema.sql`](./schema.sql)：

| 表 | 内容 |
|---|---|
| `ls_schedule` | 日程（任务 + 时间记录） |
| `ls_quick_note` | 快速记录 |
| `ls_review_answer` | 复盘答案（每题一条） |
| `ls_daily_focus` | 今日重点（自由文本，单条/日） |
| `ls_review_scheme` | 复盘方案题目 |
| `ls_daily_settlement` | 每日沉淀记录（沉淀状态机） |

### 本地 Markdown Vault（沉淀归档）

沉淀后的笔记以本地 Markdown 文件存储（不在 SQLite 中）：

| 路径 | 内容 |
|---|---|
| `Daily/<date>.md` | 当日沉淀归档（沉淀后生成） |
| `Notes/*.md` | 笔记知识库 |
| `Reviews/scheme.md` | 复盘方案 |

## 沉淀机制

### 触发

用户主动触发"沉淀今日"，或次日自动提示沉淀前一天。

### 算法

1. 读取当天所有实体（schedule / quick_note / review_answer / daily_focus）。
2. 序列化为**纯净 Markdown**（零程序标记，无 `<!-- sid -->` 等锚点）。
3. 写入本地 Vault 文件 `Daily/<date>.md`。
4. 标记实体 `settled=true`，当日数据冻结为归档，不再变化。

### 文法规范（纯净 .md）

沉淀生成的 `.md` **零程序标记**——完全人类可读，无任何隐藏注释 / 锚点 / 元数据：

```markdown
# 2026年6月23日 周一

## 今日重点
（自由文本）

## 今日日程
- 09:00-10:00 完成需求评审（工作·任务 ✓）
- ...

## 快速记录
- ...

## 今日复盘
### 今天完成了什么？
...
```

## 本地持久化

- **全部数据在本机**：SQLite 存当天热数据，Vault 目录存沉淀归档的 `.md`，无任何云端或跨设备流转。
- **单设备**：桌面端和移动端各自独立维护自己的 SQLite 与 Vault，互不同步。
- **删除即墓碑**：实体软删除用 `deleted=1`（墓碑），不做物理删除，保证归档一致性。
- **建议备份**：数据完全本地化，卸载即清除，建议定期手动备份本地数据目录。

## 回看历史

- 近期：实体 + `.md` 双轨，实体可编辑。
- 历史：纯 `.md` 归档，回看只读；如需修改，产生新的本地 Markdown 文件。
