# 开源仓库架构（单仓两公开 remote）

> 本文说明 LifeScale 开源仓库的 git 架构与发版流程。

## 架构：单代码仓 + 两公开 remote

```
                       ┌── gitee-public (gitee 公开仓) ←── 日常 git push + 发版
lifescale/ (单一代码仓) ──┤
  仓内只有 .example 模板 └── github (国际公开仓)        ←── 发版时同步 push main
  无真实密钥
```

**一份代码、两个公开 remote**，均为开源仓，内容逐字节相同，无需任何脱敏脚本。

## 为什么不需要脱敏脚本

代码仓里**从来没有真实密钥**：

| 敏感内容 | 位置 |
|---|---|
| 打包产物（APK/EXE） | `workspace/release/`（仓库目录之外） |
| 签名密钥、SSL 证书 | `workspace/secrets/`（仓库目录之外） |

`workspace/` 在 `lifescale/` 仓库目录之外，**天然不被 git 追踪**——物理隔离比规则隔离更可靠。

## Git 分支策略

本仓库是**纯本地应用**，无后端 / 无服务端 / 无需部署环境，因此不采用 GitFlow 三环境模型，简化为：

| 分支 | 角色 |
|---|---|
| `main` | 发布分支，对外可用的稳定版本 |
| `feature/*` | 需求分支，从 `main` 切出 |
| `hotfix/*` | 紧急修复，从 `main` 切出 |

没有 `staging` / `develop` 分支，也没有 staging / prod 环境之分——应用没有服务端可部署。

## 发版流程

```
feature/*（日常开发，从 main 切出）
   │ 需求完成，合入 main
   ▼
main（打 tag，可发布）
   │
   ├── git push gitee-public main    （国内公开仓，日常 + 发版）
   └── git push github main          （国际公开仓，发版同步）
```

两个公开仓内容完全一致。Gitee 用于日常 push，GitHub 在发版时同步推送 `main`。

## 红线

- ❌ 不直接在 `main` 上开发（走 `feature/*`）
- ❌ 不提交 `.env`、密钥、证书、用户数据
- ✅ 真实凭据永远只在 `workspace/secrets/`

## 验证"公众会收到什么"

最可靠的方法：`git clone` 一份自己的公开仓全新副本——那就是公众拿到的。因为没有任何"剥离步骤"，不可能遗漏敏感内容。
