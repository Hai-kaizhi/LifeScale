# 仓库架构

> 本文说明 LifeScale 开源仓库的 git 架构与发版流程。

## 架构：单代码仓 + 两公开 remote

```
                  ┌── gitee-public（Gitee 公开仓）  ← 日常 push + 发版
本仓库 ───────────┤
                  └── github（GitHub 公开仓）       ← 发版时同步 push main
```

**一份代码、两个公开 remote**，内容逐字节相同。两个 remote 均为公开开源仓，公众 clone 任意一个即得到完整源码。

> 本仓库是**纯本地应用**：无后端、无服务端、无密钥配置、无部署环境，因此仓库内不存在任何敏感信息，无需脱敏处理。

## Git 分支策略

应用是纯本地单机应用，无需部署多环境，因此不采用 GitFlow 三环境模型，简化为：

| 分支 | 角色 |
|---|---|
| `main` | 发布分支，对外可用的稳定版本 |
| `feature/*` | 需求分支，从 `main` 切出 |
| `hotfix/*` | 紧急修复，从 `main` 切出 |

没有 `staging` / `develop` 分支，也没有 staging / prod 环境之分。

## 发版流程

```
feature/*（日常开发，从 main 切出）
   │ 需求完成，合入 main
   ▼
main（打 tag，可发布）
   │
   ├── git push gitee-public main    （Gitee 公开仓，日常 + 发版）
   └── git push github main          （GitHub 公开仓，发版同步）
```

两个公开仓内容完全一致。Gitee 用于日常 push，GitHub 在发版时同步推送 `main`。

## 红线

- ❌ 不直接在 `main` 上开发（走 `feature/*`）
- ❌ 不提交打包产物（APK/EXE）、签名密钥、用户数据
- ✅ 打包产物输出到 `dist/release/`（已被 `.gitignore` 忽略）
