# LifeScale 技术架构

> 本文是 LifeScale **开源本地版**的技术架构说明。纯本地应用，不含任何后端 / 云同步 / 数据库服务端 / 部署编排。

## 一句话

**本地优先（Local-First）的 Markdown Vault 单机应用**：本地 SQLite 是当日结构化数据的真相源，沉淀后产出纯净 Markdown 文件。桌面端 / 移动端各自独立运行、独立存储，单设备、无同步。

## 核心模型

### 本地优先 + SQL-first 真相源

- **当日结构化数据**（日程 / 快记 / 复盘 / 今日重点）存本地 SQLite，离线完整可用。
- **沉淀机制**：当天数据可"沉淀"为纯净 Markdown 文件（零程序标记），转入本地 Vault 目录。
- **本地是唯一真相源**：没有云端镜像，没有跨设备同步。每台设备各自维护自己的数据。

### 两层存储

| 层 | 载体 | 职责 |
|---|---|---|
| 本地业务库 | SQLite（`lifescale.db`） | 当日结构化实体真相源（6 表） |
| 本地 Vault | 本地 Markdown 文件（`Daily/*.md`、`Notes/*.md`、`Reviews/scheme.md`） | 沉淀后的纯净归档，冷数据 |

**附件**（图片等）按内容哈希（SHA-256）缓存在本地数据目录，去重存储，不进入 SQLite。

### 沉淀（沉淀 / Settlement）

- **当天**：结构化实体在 SQLite 中频繁 CRUD（热数据、可变）。
- **沉淀后**：当天实体序列化为**纯净 Markdown**（零程序标记），写入本地 Vault 的 `Daily/<date>.md`，成为冷归档（不可变）。
- 沉淀后实体标记 `settled=true`，不再变化；归档 `.md` 即历史真相。

## 两端技术栈

| 端 | 技术 | 包管理 |
|---|---|---|
| 桌面端 | Tauri 2 + React 19 + TypeScript + Vite + Ant Design | pnpm |
| 移动端 | Flutter + Riverpod + go_router | pub |
| 本地数据库 | SQLite（桌面 rusqlite / 移动 sqflite） | - |

> 无后端、无 PostgreSQL、无 Redis、无 Docker。

## 数据库 Schema

详见 [`database/schema.sql`](./database/schema.sql)（单一真相源）。

- 本地 SQLite：6 张表（4 Daily 实体 + 2 本地独有：复盘方案、沉淀记录）。
- Vault 文件（Daily / Notes / Reviews）以本地 Markdown 文件存储，不在 SQLite 中。
